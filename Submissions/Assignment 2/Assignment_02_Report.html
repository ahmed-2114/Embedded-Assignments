Assignment

Assignment 02: Wrong Stack Pointer Restoration Demo

Objective

The objective of this assignment is to demonstrate what happens when a task context is restored using the wrong stack pointer. Instead of restoring the correct saved exception frame, the program intentionally shifts the stack pointer by one word for Task 2. This causes the decoded program counter and status register fields to become invalid, which is then detected by the corruption check.

System Design

The program builds two synthetic Cortex-M exception frames, one for Task1() and one for Task2(), using BuildInitialFrame(). A simulated context-restore routine named AttemptContextRestore() is called periodically from SysTick_Handler().

The restore routine selects the next task, chooses the correct saved stack pointer in g_restoredSp, intentionally shifts it for Task 2 using g_faultyRestoreSp = g_restoredSp + 1, decodes the frame, and checks whether the decoded pc and xpsr are valid.

Variables Used for Verification

The main watch variables used during debugging were g_ui32SysTickCount, g_restoreAttemptCount, g_currentTaskId, g_nextTaskId, g_contextCorruptionDetected, g_expectedPc, g_observedPc, g_faultSignature, g_currentSp, g_restoredSp, g_faultyRestoreSp, g_lastDecodedFrame.pc, and g_lastDecodedFrame.xpsr.

Debug Method

Two breakpoints were used inside AttemptContextRestore(): one at the start of the restore attempt and one at the corruption-check if condition. The Memory windows were used to compare g_restoredSp and g_faultyRestoreSp.

Experimental Result Summary

The debugger results show a complete cause-and-effect chain: the correct saved frame exists at g_restoredSp, the restore address is intentionally changed to g_faultyRestoreSp = g_restoredSp + 1, decoding from that shifted address corrupts the frame fields, the decoded pc no longer matches g_expectedPc, the decoded xpsr becomes invalid, and the corruption flag is raised.

Conclusion

The assignment objective was achieved successfully. The implementation demonstrates that restoring a task context from the wrong stack pointer causes corrupted exception-frame decoding. The debugger evidence clearly shows the pointer shift, the resulting mismatch between expected and observed program counter values, and the final corruption detection flag.
